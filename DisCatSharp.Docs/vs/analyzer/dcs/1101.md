# DisCatSharp Analyzer Rule DCS1101

This rule identifies code that filters `DiscordClient.Presences` manually for a single user.

The modern API exposes `DiscordClient.GetPresences(ulong userId)` for this scenario.

## What the code fix handles

The current code fix is intentionally conservative.

It handles patterns such as:

- `client.Presences.Values.Where(p => p.UserId == userId)`
- `client.Presences.Values.First(p => p.User.Id == userId)`
- `client.Presences.Values.FirstOrDefault(p => p.User.Id == userId)`
- `client.Presences.Values.Single(p => p.User.Id == userId)`
- `client.Presences.Values.SingleOrDefault(p => p.User.Id == userId)`
- `client.Presences.Values.Any(p => p.User.Id == userId)`
- `client.Presences.Where(kvp => kvp.Value.UserId == userId).Select(kvp => kvp.Value)`

When one of these shapes is detected, the fixer rewrites the expression to:

- `client.GetPresences(userId).Values`

For direct lookup shapes such as `First(...)` or `Any(...)`, the fixer preserves the terminal LINQ call and removes the now-redundant predicate.

This keeps downstream LINQ such as `.ToList()` intact while switching the source expression to the dedicated API.

## Why this rule exists

`DiscordClient.Presences` only represents the aggregate presence cache. `GetPresences(userId)` returns the cached presences for that user keyed by guild id and includes the aggregate entry when it exists.

Using `GetPresences(userId)` makes the intent clearer and avoids ad-hoc filtering against the aggregate cache.
